# raglib — план реализации RAG-библиотеки

> Рабочее имя пакета — `raglib` (легко переименовать на этапе 0).

> **Статус (2026-07-07): этапы 0–7 реализованы.** Код в `src/raglib/`, тесты в
> `tests/` (41 тест, все офлайн: HashingEmbeddings + MockLLM + фикстуры),
> wheel собирается (`dist/`). Из этапа 8 сделаны `integrations/langchain_tools.py`
> (обёртки search/toc/read_section/agentic); остальное этапа 8 — по мере надобности.

## 1. Цель и границы

Самостоятельный Python-пакет, который:

1. **Принимает на вход** файл или набор файлов (путь/пути/директория).
2. **Прогоняет через «распознавание текста»** — в проде это делает целевая
   система (она отдаёт `.md`); в библиотеке распознавание — это интерфейс
   с мок-реализацией. Контракт: *файл на входе → markdown на выходе*.
3. **Строит персистентный иерархический индекс** (папка на диске) с полным
   жизненным циклом: создать / сохранить / загрузить / **удалить**.
4. **Даёт поиск несколькими инструментами** над одним индексом:
   - BM25 (лексический, ранжированный);
   - векторный (FAISS, косинусная близость по эмбеддингам) — плоский по чанкам
     или иерархический coarse-to-fine (сначала разделы, потом чанки);
   - гибридный (BM25 + вектор через Reciprocal Rank Fusion);
   - навигация по оглавлению: получить outline документа, выбрать раздел по
     номеру/заголовку/смыслу и прочитать его целиком (включая подразделы);
   - regex/grep (дёшево, без индекса; выдача — те же целые пункты);
   - **агентский поиск**: свободный промт → детерминированный цикл
     «план → поиск всеми инструментами → LLM-рефлексия релевантности →
     дообыск» (см. §6).

**Инвариант выдачи:** результат поиска — всегда **целый пункт документа вместе
с его номером** («12.1.4. <полный текст>»), никогда не обрезанный на середине
фрагмент. Выдача обрабатывается программно на стороне потребителя, поэтому
номер пункта и полнота текста — обязательная часть контракта `SearchHit`,
а не best-effort.

**Не входит в задачу:** сам OCR (мок), LLM-генерация финального ответа
(агентский поиск использует LLM только для планирования и оценки
релевантности — §6; синтез ответа остаётся потребителю), UI, внешние БД/сервисы.

## 3. Архитектура

### Поток данных

```
файлы (md/pdf/docx/скан)
   │
   ▼
TextRecognizer (интерфейс)          ← в проде: адаптер целевой системы
   │  MockRecognizer: .md/.txt — passthrough, остальное — ошибка/фикстура
   ▼
markdown-текст  ──► MarkdownParser ──► Document { sections: дерево по нумерации }
   │
   ▼
ClauseSegmenter (единица — ПУНКТ: нумерация заголовков И нумерованных абзацев;
                 пункт никогда не режется)
   │
   ▼
Clause { doc_id, clause_key, section_path, text, span }    ← единица ВЫДАЧИ
   │
   ▼  длинный пункт → несколько окон-юнитов; короткий → один юнит (1:1)
Chunk { unit_id → clause_id, text }                        ← единица ИНДЕКСА
   │
   ├─► BM25Index (rank_bm25 по юнитам, пересборка при загрузке)
   ├─► VectorIndex (FAISS, два уровня: chunks.faiss + sections.faiss;
   │                кеш эмбеддингов по sha1 контента)
   └─► TocIndex (дерево разделов всех документов — скелет иерархии)
   │
   ▼
IndexStore — папка на диске (создание/загрузка/удаление):
   manifest.json   (версия формата, список документов, параметры чанкинга,
                    модель эмбеддингов, размерность, faiss index_factory)
   clauses.jsonl   (ПУНКТЫ — единицы выдачи: номер + полный текст;
                    № строки = clause_id)
   chunks.jsonl    (индекс-юниты со ссылкой на clause_id;
                    № строки = id вектора в chunks.faiss)
   sections.jsonl  (узлы дерева разделов; № строки = id в sections.faiss)
   chunks.faiss    (FAISS-индекс векторов юнитов; отсутствует в BM25-only режиме)
   sections.faiss  (FAISS-индекс векторов разделов — верхний уровень иерархии)
   embed_cache/    (sha1-кеш сырых эмбеддингов .npz: пересборка индекса
                    не дергает эмбеддер повторно; удаляется вместе с индексом)
   toc.json        (оглавления документов)
   docs/<id>.md    (распознанные markdown-тексты — нужны для read_section)
   │
   ▼
SearchEngine — единая точка поиска: bm25 / vector (flat|tree) / hybrid / toc / grep,
фильтры по документу и по префиксу раздела; юниты агрегируются до пунктов,
результат — SearchHit (целый пункт + его номер)
   │
   ▼
AgenticSearcher (опционально, поверх SearchEngine): промт → план → поиск всеми
инструментами → LLM-рефлексия релевантности → дообыск; выдача та же — целые
пункты + трасса шагов (см. §6)
```

## 4. Публичный API (целевой вид)

```python
from raglib import RagIndex
from raglib.recognition import MockRecognizer
from langchain_gigachat import GigaChatEmbeddings   # любой LangChain-эмбеддер, или None

# 1) Построение индекса из файлов (на тестах — .md, как отдаёт целевая система)
index = RagIndex.build(
    inputs=["docs/charter.md", "docs/"],          # файл, список, директория
    index_dir="./charter_index",
    recognizer=MockRecognizer(),                   # шов для реальной системы распознавания
    embeddings=GigaChatEmbeddings(...),            # объект LangChain; None → BM25-only индекс
    chunk_size=1500, chunk_overlap=150,            # параметры ВНУТРЕННИХ окон
)

# 2) Загрузка готового индекса (без пересборки)
index = RagIndex.load("./charter_index")

# 3) Поиск разными инструментами
hits = index.search("крупные сделки", method="hybrid", top_k=5)   # bm25|vector|hybrid
hits = index.search("крупные сделки", method="vector",
                    strategy="tree")               # coarse-to-fine: разделы → пункты
hits = index.search("одобрение сделок", method="bm25",
                    section="12", doc="charter")                   # фильтры
hits = index.grep(r"\d+\s*%\s*балансовой стоимости")               # regex; выдача — пункты

# 4) Оглавление и разделы
print(index.toc())                          # outline всех документов
print(index.toc(doc="charter"))             # одного документа
sec = index.find_section("Наблюдательный совет")               # по заголовку
sec = index.find_section("кто одобряет крупные сделки",
                         semantic=True)     # по смыслу — через sections.faiss
text = index.read_section("charter", "12.1")       # раздел целиком, с подразделами

# 5) Агентский поиск: промт → цикл с рефлексией (см. §6)
from langchain_gigachat import GigaChat     # любая LangChain chat-модель (.invoke)
llm = GigaChat(credentials="...", scope="GIGACHAT_API_CORP", verify_ssl_certs=False)
res = index.agentic_search(
    "Какие сделки требуют одобрения наблюдательного совета и с какими порогами?",
    llm=llm, top_k=8, max_iters=3,
)
res.hits       # те же SearchHit: целые пункты с номерами
res.trace      # шаги: запросы, инструменты, вердикты рефлексии, решения
res.degraded   # True, если цикл падал в обычный hybrid (LLM недоступна/непарсибельна)

# 6) Выдача — всегда целые пункты с номерами, пригодные для программной обработки
for h in hits:
    print(h.clause_number, h.score, h.doc_name)   # "12.1.4" 3.38 "charter.md"
    print(h.breadcrumb)   # "Статья 12. … › 12.1. Компетенция… › 12.1.4"  (глава→пункт)
    print(h.section_path) # ["12", "12.1", "12.1.4"]  — числовой путь (стабильный API)
    print(h.text)         # ПОЛНЫЙ текст пункта — никогда не обрезается

# 7) Удаление индекса: валидация, что это индекс raglib, затем удаление папки
RagIndex.delete_index("./charter_index")    # или index.delete() на загруженном
```

Ключевые свойства API: **все инструменты работают над одним индексом**; каждый
`SearchHit` несёт `clause_number` + полный текст пункта, а также провенанс —
`doc_name` и путь глава→пункт (`section_path` числами; `section_titles`/`breadcrumb`
с заголовками); внутренние окна агрегируются и дедуплицируются до пункта ещё в
движке; цепочка «нашёл пункт → прочитал его раздел целиком» — это один вызов
`read_section`. Полный контракт выдачи — в [README](README.md#контракт-выдачи).

## 5. Иерархический индекс (FAISS): как формируется

Иерархия — доменная: **корпус → документ → дерево разделов → пункты**.
Она строится один раз при `build()` и сохраняется целиком в папке индекса.

```
Уровень 1: документ         doc_id, метаданные             → manifest.json
Уровень 2: разделы          дерево по нумерации            → sections.jsonl + sections.faiss
                            (Статья 12 ⊃ 12.1 ⊃ 12.1.4)
Уровень 3: пункты (листья)  целый пункт с номером;         → clauses.jsonl
                            единица ВЫДАЧИ, не режется
Уровень 4: окна-юниты       только внутри длинных пунктов; → chunks.jsonl + chunks.faiss
                            единица ИНДЕКСА, наружу не отдаются
```

### Шаги формирования

1. **Скелет — дерево разделов.** Markdown-парсер строит дерево по **нумерации
   заголовков**, а не по уровню `#` (распознавание сплющивает уровни — это уже
   проверено на реальных уставах в Deep agent): `Статья 12` ⊃ `12.1` ⊃ `12.1.4`.
   Узел: `(doc_id, key, title, span, parent, children)`. Дерево сериализуется
   в `sections.jsonl` (плоский список узлов со ссылками) и `toc.json` (outline).

2. **Листья — пункты (единица выдачи).** Тело каждого раздела сегментируется
   на пункты по нумерации: и заголовки, и **нумерованные абзацы** (строки,
   начинающиеся с `12.1.`, `12.1.4.`, `12.1.4)` …) — в распознанном markdown
   пункт чаще всего не заголовок, а обычный абзац. Пункт **атомарен**: не
   режется ни при индексации, ни в выдаче; вложенные перечисления без своей
   сквозной нумерации (а), б), 1)…) остаются внутри родительского пункта.
   Ненумерованный текст (преамбула и т.п.) образует пункт с ключом ближайшего
   раздела. Пункты пишутся в `clauses.jsonl` (№ строки = `clause_id`).

3. **Окна-юниты — единица индекса.** Пункт, превышающий лимит контекста
   эмбеддера / `chunk_size`, для индексации режется окном 1500/150 на
   несколько юнитов; короткий пункт — ровно один юнит. Каждый юнит ссылается
   на свой `clause_id`. Юниты пишутся в `chunks.jsonl`; номер строки = позиция
   вектора в FAISS (implicit ordinal id — достаточно, пока индекс
   пересобирается целиком). Наружу юниты не отдаются никогда.

4. **Векторы — снизу вверх (bottom-up).**
   - *Юниты* эмбеддятся напрямую (GigaChat батчами; sha1-кеш в `embed_cache/`,
     поэтому пересборка индекса не вызывает эмбеддер повторно).
   - *Раздел* по умолчанию получает вектор как **L2-нормированное среднее
     векторов своих потомков** (mean-pooling юнитов и подразделов, снизу вверх
     по дереву; пустые разделы в sections.faiss не попадают). Ноль
     дополнительных запросов к эмбеддеру — существенно для закрытого контура.
     Опция `section_embedding="title+lead"` — эмбеддить «заголовок + первые N
     символов» (+1 запрос на раздел, качественнее для семантического выбора
     раздела) — включается флагом, сравним на фикстурах.
   - *Документ* (опционально) — среднее векторов его разделов; нужен только
     для роутинга по большому многодокументному корпусу.

5. **Два FAISS-индекса.** `chunks.faiss` и `sections.faiss` — оба
   `IndexFlatIP` поверх L2-нормированных векторов (внутреннее произведение =
   косинус). Это **точный** поиск: без обучения и гиперпараметров, на целевых
   объёмах (10³–10⁵ векторов) — миллисекунды. Сохранение —
   `faiss.write_index()` в папку индекса. Тип индекса — строка `index_factory`
   в `manifest.json`: если корпус вырастет за ~10⁵ юнитов, меняем строку на
   `"HNSW32,Flat"` — формат хранилища и код поиска не меняются.

6. **Поиск по иерархии — `strategy="tree"` (coarse-to-fine):**
   1. один эмбеддинг запроса;
   2. `sections.faiss` → top-s релевантных разделов (s≈8);
   3. `chunks.faiss`, суженный до юнитов этих разделов (`IDSelectorBatch` в
      search params; fallback — post-filter с over-fetch, выберем по версии
      faiss в контуре) → кандидаты-юниты;
   4. **агрегация юнитов до пунктов**: score пункта = max по его юнитам,
      дубликаты одного пункта схлопываются → top-k ЦЕЛЫХ пунктов с номерами;
   5. каждый hit несёт путь в дереве (`12.1.4 → 12.1 → 12`), так что следующий
      шаг потребителя — `read_section` нужного предка.

   По умолчанию `strategy="flat"` — плоский поиск по `chunks.faiss`: на малом
   корпусе он точнее и проще. BM25 всегда плоский по юнитам; агрегация до
   пунктов и RRF-слияние в гибриде идут по `clause_id`, поэтому обе стратегии
   совместимы с гибридом и дают одинаковую форму выдачи — целые пункты.
   `sections.faiss` дополнительно обслуживает `find_section(..., semantic=True)` —
   «выбрать раздел по оглавлению по смыслу», без чтения пунктов.

**Терминологическая оговорка.** HNSW в FAISS — тоже «иерархический», но это
внутренняя иерархия графа навигации (ускорение ANN-поиска). Здесь же иерархия
доменная — документ → раздел → пункт (качество и объяснимость результатов).
Первая — опция при росте объёма (шаг 5), вторая строится всегда.

## 6. Агентский поиск: промт → цикл с рефлексией

Вход — свободный промт («какие сделки требуют одобрения наблюдательного
совета?»), выход — отобранные **целые пункты с номерами** (инвариант §1
сохраняется) плюс трасса шагов. Финальный ответ по пунктам синтезирует
потребитель — библиотека отвечает за то, чтобы в выдаче было всё нужное и
ничего лишнего.

**Главное проектное ограничение — слабая модель.** В Deep agent уже проверено
на minimax-m3: free-form агент, который сам решает, какие тулы звать,
разваливается (пропускает шаги, зовёт finish на первом ходу). Поэтому агент
здесь — **не LLM с тулами, а детерминированный цикл в коде**, где LLM
выполняет три маленькие узкие задачи с толерантным парсингом JSON — тот же
паттерн, что в `extraction.py`, стабильный на этой модели. Библиотека при этом
не запрещает сильную модель — цикл одинаково работает с любой.

### Устройство цикла

```
промт пользователя
   │
   ▼
[1] PLAN (LLM-вызов №1): промт → 1–3 поисковых запроса-переформулировки
      (+ аспекты, которые нужно покрыть)
      Маршрутизация инструментов — ЭВРИСТИКИ В КОДЕ, не LLM:
      номера/проценты/кавычки → bm25 + grep; концептуальные формулировки →
      vector; «в каком разделе…» → find_section(semantic=True)
   │
   ▼
[2] RETRIEVE (код, без LLM): для каждого запроса — hybrid (BM25+vector, RRF)
      + выбранные эвристиками инструменты; объединение кандидатов,
      дедупликация по clause_id (кандидатов больше, чем top_k: over-fetch ×3)
   │
   ▼
[3] REFLECT (LLM-вызов №2): батч-оценка кандидатов (до ~10 пунктов за вызов):
      [{clause: "12.1.4", verdict: relevant|partial|irrelevant,
        missing: "какого аспекта не хватает"}, …]
      В промт оценки пункт кладётся усечённым (~800 симв.) — усечение ТОЛЬКО
      внутри промта, выдача всегда целые пункты. Вердикты кешируются по
      (аспект, clause_id) — пункт не оценивается дважды.
   │
   ▼
[4] DECIDE (код): каждый аспект из PLAN покрыт ≥1 relevant-пунктом
      или набрано ≥ top_k relevant?  → СТОП, вернуть результат.
      Иначе → [5] REFINE (LLM-вызов №3): переформулировать запросы по missing
      (может предложить и смену инструмента — например, прочитать конкретный
      раздел через toc) → снова [2]. Максимум max_iters (по умолчанию 3).
   │
   ▼
Результат: hits = relevant + partial (сортировка: вердикт, затем RRF-score),
trace = полный журнал шагов, degraded = False
```

### Гарантии и деградация

- **Жёсткие бюджеты**: `max_iters` (3), `max_llm_calls` (≈8), кандидатов на
  итерацию ≤ 30; grade-кеш внутри запуска. Худший случай предсказуем по
  токенам и времени.
- **Никогда не хуже обычного поиска**: любой сбой LLM (таймаут, 429,
  непарсибельный ответ после толерантного парсинга) → шаг пропускается, и
  метод честно возвращает hybrid top-k с `degraded=True` и причиной в trace.
  Агентский поиск не добавляет новых способов упасть.
- **`llm=None` → чистый hybrid**: удобно для A/B-сравнения качества с
  рефлексией и без на одних и тех же запросах.
- **LLM — объект LangChain напрямую, не зависимость**: агентский цикл зовёт
  `llm.invoke(messages) -> AIMessage` (структурный протокол `ChatModel`) и сам
  разворачивает контент в текст (`content_to_text`, в т.ч. content-блоки).
  Никаких raglib-адаптеров: передавайте `langchain_gigachat.GigaChat`,
  `ChatOpenAI` или модель из своего deepagents-стека как есть; `MockLLM` со
  сценарием ответов — для тестов. Ядро по-прежнему без langchain в зависимостях.
- **Трасса обязательна**: `res.trace` фиксирует запросы, инструменты,
  кандидатов, вердикты и решения каждого шага — это и отладка качества, и
  материал для приёмки/аудита в контуре.

### Предложения по реализации (v1 vs потом)

- **В v1**: multi-query из PLAN (дёшево, заметно поднимает recall);
  verdict-приоритет в сортировке (relevant > partial); покрытие аспектов как
  критерий останова; grade-кеш; полная трасса.
- **Сознательно НЕ в v1**: HyDE и LLM-реранкер поверх рефлексии — на слабой
  модели выигрыш сомнителен, а бюджет вызовов растёт кратно; замерим на
  фикстурах после ядра (этап 8). Кросс-энкодерный реранкер — это локальная
  модель, в контур её нести отдельная задача.

## 7. Ключевые проектные решения

1. **Зависимости — минимальные**: `numpy`, `rank-bm25`, `faiss-cpu`;
   LLM и эмбеддинги — готовые объекты LangChain, которые передаёт вызывающий код
   (провайдер ставится отдельно: `[gigachat]` и т.п.); `langchain-core` только в
   extra `[langchain]` (для `@tool`-обёрток). Никакого
   docling/langchain в ядре — библиотека не парсит PDF сама (это делает
   целевая система), поэтому ядро ставится офлайн из wheels. Модели данных —
   stdlib `dataclasses`, не pydantic.
2. **Хранение — папка на диске, без сервисов** (закрытый контур: см. п. 10).
   Векторы — `chunks.faiss`/`sections.faiss` через `faiss.write_index`; BM25 не
   сериализуем (pickle хрупок и небезопасен), а пересобираем из `chunks.jsonl`
   при `load()` — на целевых объёмах это миллисекунды.
3. **Единица выдачи — пункт целиком, с номером (small-to-big retrieval)**:
   сегментация по нумерации заголовков И нумерованных абзацев; матчинг может
   идти по внутренним окнам-юнитам, но наружу движок отдаёт только целый
   пункт + `clause_number` (агрегация max-score, дедупликация по `clause_id`).
   Обрезки выдачи нет — потребитель обрабатывает пункты программно и полагается
   на их полноту; фильтры и иерархия работают через `section_path` пункта.
4. **Удаление индекса — только с валидацией**: `RagIndex.delete_index(dir)` и
   `index.delete()` сначала проверяют, что папка действительно индекс raglib
   (валидный `manifest.json` нашей версии формата), закрывают FAISS-хендлы и
   только потом удаляют папку целиком (включая `embed_cache/`). На «не-индекс» —
   отказ с ошибкой: никакого rmtree по произвольному пути. `index.delete()`
   переводит объект в закрытое состояние (дальнейшие вызовы — ошибка).
5. **Агентский поиск — цикл в коде, LLM только в трёх точках** (PLAN /
   REFLECT / REFINE), с бюджетами и деградацией в hybrid (§6). Урок Deep agent:
   на слабой модели нельзя отдавать оркестрацию модели.
6. **Токенизация RU**: `\w+` + lower + нормализация словоформ,
   `bm25_normalizer="none"|"stem"|"lemma"`. Режим фиксируется в manifest
   (запросы нормализуются так же, как корпус), при `load()` переопределяется
   без пересборки (BM25 всё равно строится в памяти). **Замер на реальных
   уставах (два устава ООО/АО, 8 перефразированных запросов, 2026-07-07):**
   bm25/none MRR 0.719 → bm25/stem **0.917** → bm25/lemma 0.854; Σрел@5
   24 → 34 → 30; контрольный пункт 1.6 (неприменение ст. 45) поднялся из
   «вне топ-10» на ранг 1 в обоих режимах. Итог: **stem — дефолт**
   (snowballstemmer перенесён в ядро, чисто-python), lemma — extra `[ru]`.
7. **Гибрид — RRF** (k=60): не требует нормализации несравнимых скор BM25 и
   косинуса, устойчив, 10 строк кода; слияние — по `clause_id`.
8. **Мок распознавания повторяет контракт целевой системы**: вход — путь к
   файлу, выход — markdown-строка. `MockRecognizer` пропускает `.md`/`.txt`
   как есть; для остальных расширений — понятная ошибка «в проде это делает
   система X». `FixtureRecognizer(mapping)` — для тестов интеграции
   (`file.pdf → заранее подготовленный file.md`).
9. **Эмбеддинги — свой протокол из двух методов** (`embed_documents`,
   `embed_query`), сигнатурно совместимый с LangChain — любой langchain-эмбеддер
   подойдёт без обёрток, но зависимости на langchain нет. Кеш по sha1
   содержимого юнитов сохраняем (перенос из `search_vector`).
10. **`RagIndex.build(embeddings=None)` — законный режим**: BM25 + TOC + grep
    работают полностью офлайн, `.faiss`-файлы просто не создаются;
    `method="vector"|"hybrid"` в этом режиме даёт понятную ошибку. Это же —
    деградация при недоступности GigaChat.

## 9. Тестирование

- **Unit:** парсер секций (нумерация, вложенность, edge-cases); сегментация
  пунктов (пункты-абзацы, вложенные перечисления, ненумерованная преамбула);
  окна-юниты (длинный пункт → несколько юнитов → один hit целым пунктом);
  RRF; BM25-токенизация; mean-pooling разделов (нормировка, пустые разделы);
  store (roundtrip save/load, включая `faiss.write_index`/`read_index`;
  отказ на чужой версии manifest); delete (валидация: отказ на произвольной
  папке, идемпотентность повторного вызова).
- **Инварианты выдачи (сквозные, для всех методов, включая агентский):**
  каждый `SearchHit` несёт `clause_number`; текст hit'а — целый пункт,
  посимвольно равный сегменту исходного markdown (никогда не обрезан); один
  пункт не встречается в выдаче дважды.
- **Агентский цикл (на `MockLLM`, офлайн):** happy-path; refine добирает
  пропущенный пункт; непарсибельная LLM → `degraded=True`, результат равен
  hybrid; соблюдение бюджетов `max_iters`/`max_llm_calls`; grade-кеш не даёт
  повторных оценок одного пункта.
- **Integration:** build → save → load → все методы поиска, обе стратегии
  (flat/tree); только офлайн (`HashingEmbeddings`, `FixtureRecognizer`,
  `MockLLM`) — весь пайплайн покрывается без сети, никакой сети в CI.
- **Контрактные:** `TextRecognizer` (мок ↔ будущий адаптер); `EmbeddingsLike`
  и `ChatModel` — проверяем, что duck-typed объект LangChain (`.embed_*`,
  `.invoke`) ведёт сборку и агентский цикл без адаптеров (`HashingEmbeddings`,
  `MockLLM`).
- Тестовые данные — только синтетические фикстуры; реальные уставы из
  `input/` в пакет и тесты не тащим (конфиденциальность).

## 10. Ограничения целевой среды (закрытый контур)

- Установка офлайн: ядро — Python + numpy/rank-bm25 + `faiss-cpu` (бинарный
  manylinux-wheel, заносится в контур вместе с остальными, pin версии
  ≥1.7.4); никаких загрузок моделей/данных в рантайме.
- Эмбеддинги — только GigaChat через доступный шлюз Sber cloud
  (адаптер уже учитывает Минцифры CA и статический токен для on-prem)
  либо BM25-only режим.
- LLM для агентского поиска — корпоративный OpenAI-совместимый шлюз
  (minimax-m3); дизайн цикла рассчитан именно на слабую модель (§6),
  сильная модель — бонус, не требование.
- Никаких внешних сервисов (БД, elasticsearch) — индекс это папка на диске,
  FAISS работает in-process.
- Документы русскоязычные — токенизация, промты агента и фикстуры под RU.
